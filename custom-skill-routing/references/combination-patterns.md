# Combination Patterns

Use these only when one owner alone would hide an important dependency or quality check.

- Ambiguous feature request:
  `product-brainstorming` -> `spec-driven-development`
- Existing design that needs a written PRD:
  `figma` or `google-stitch` -> `design-to-prd`
- Ticket-board sprint planning:
  `ticket-batch-intake` -> `spec-driven-development` or `task-decomposition-and-resume`
- Architecture refactor shaping:
  `architecture-deepening-audit` -> `grill-with-docs` -> `test-driven-development` or `spec-driven-development`
- Disposable exploration before commitment:
  `throwaway-prototype` -> `product-brainstorming` or `spec-driven-development`
- Planned delivery already underway:
  `task-decomposition-and-resume` + `docs-driven-execution`
- Interrupted or transferred session:
  `session-handoff` for a fresh continuation note, or `artifact-resume-audit` first when old artifacts may be stale
- Multi-screen app build from Figma or Stitch:
  source adapter + `design-to-traversable-app`, then `docs-driven-execution`
- Existing frontend plus backend contract work:
  `integrating-backend-api-into-frontend` + `traceable-delivery`
- E2E handoff for another agent:
  `e2e-verification-handoff` -> `playwright` or Browser execution -> `verification-before-completion` -> `release-readiness`
- Review before closeout:
  `two-axis-review` -> `verification-before-completion` -> `release-readiness`
- Final closeout before handoff:
  `release-readiness`, then `reflection-and-learning` when the lessons should survive
- Bug discovered during delivery:
  `systematic-debugging`, then return to the current owner or escalate to `spec-driven-development` if the issue is really source drift
